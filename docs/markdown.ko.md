# 마크다운 노트로 ADDE 사용하기

버튼이 없는 환경에서는 **마크다운 노트 파일 편집만으로** AI 레인을 구동합니다. 지시는 인박스 노트에 체크박스로 보내고, 권한은 승인 노트의 체크박스로 허용/거부하며, 응답은 출력 노트로 받습니다.

이 방식은 임의의 마크다운 에디터에서 **로컬 파일만으로** 동작합니다. 모바일에서 노트를 보려면 **이미 쓰는 동기화 도구(Obsidian Sync·Syncthing·iCloud 등) 아무거나**로 폴더를 동기화하면 되고, ADDE 는 어떤 도구를 쓰든 상관하지 않습니다. 아래 예시는 Obsidian 을 쓰지만 Obsidian 전용 내용은 없습니다.

## 동작 개요

```
[inbox 노트]          --(send 체크박스)-->  ADDE  --(ACP)-->  AI 엔진(claude 등)
[approvals/<req>.md]  <--(권한 요청, 요청당 파일)-- ADDE
[approvals/<req>.md]   --(allow/deny 체크)-->  ADDE (게이트 반영)
[out/ 노트]           <--(응답 1메시지=1파일)-- ADDE
```

- 한 레인 = `(마크다운 노트 ↔ 프로젝트 폴더)` 한 쌍. 여러 레인을 등록해 여러 메모·폴더를 각각 매핑할 수 있습니다.
- 푸시 알림은 없습니다(파일 기반). 노트를 열어두고 보는 **능동 세션**을 전제로 합니다.

## 1. 레인 설정

레인은 설정 파일 1개 = 레인 1개입니다. `~/.config/adde/<proj>/lanes.d/<lane>.conf` 에 작성합니다(`<proj>`·`<lane>` 은 임의 이름).

```ini
source=markdown
backend=acp
engine=claude-agent-acp
perm_tier=acp
acp_version=v1

# AI 엔진이 실제로 작업할 프로젝트 폴더(절대경로)
cwd=/Users/me/work/my-project

# markdown 어댑터 키는 markdown.<field> 네임스페이스
# 마크다운 루트 디렉터리(절대경로, 예: Obsidian vault)
markdown.root=/Users/me/ObsidianVault

# root 상대 경로 — 입력 노트(필수)
markdown.inbox=adde/my-lane/inbox.md

# 선택(미지정 시 inbox 형제로 자동): 승인 디렉터리(요청당 파일) / 출력 디렉터리
markdown.approvals=adde/my-lane/approvals/
markdown.outbox=adde/my-lane/out/

# 선택(옵트인): 아카이브 디렉터리. 지정 시 전송 시점에 메시지 본문을 이 디렉터리 하위 날짜별
# 파일(YYYY-MM-DD.md)로 이관하고 inbox 엔 짧은 `✅ sent [[...]]` 마커만 남긴다(상주 inbox 비대화 방지).
# 미지정 시 본문은 inbox 에 잔존(현행 동작) — `🗄️ archive` 체크로 필요할 때 수동 이관 가능.
# 참고: v0.1.5 이전엔 파일 하나였다. 구버전에서 쓰던 기존 단일 아카이브 파일은 업그레이드 후 레인을
# 처음 기동할 때 자동 이관된다(아래 백업 폴더가 설정돼 있으면 그 폴더로, 아니면 `<이름>.legacy` 로
# 곁에 보존).
markdown.archive=adde/my-lane/sent-archive/

# 선택(옵트인, 기본 off): 로컬 백업 폴더. 이관 기준일(retention_days)보다 오래된 출력노트·결정된
# 승인·아카이브 파일을 하루에 한 번 이 폴더로 옮긴다 — 자세한 내용은 아래 "vault 가볍게 유지하기" 참고.
# markdown.backup=/Users/me/adde-backup
# markdown.retention_days=2

# 선택(옵트인, 기본 off): 완료 후 이만큼 지난 내부 관리 파일도 함께 삭제(vault 노트 아님 —
# 아래 "vault 가볍게 유지하기" 참고).
# markdown.out_retention_days=5

# 선택: vault 의 동기화 제공자(local | icloud). 위 markdown.backup 을 쓸 때만 의미가 있다.
# markdown.sync_provider=icloud

# 선택: 자주 쓰는 도구를 미리 허용해 승인 빈도 축소(게이트는 유지)
allowlist=Read,Grep

# 선택(옵트인): denylist 외 전부 자동 허용 — denylist 도구·패턴만 승인 노트가 생성됨
# (autopass 에서 denylist 생략 시 내장 기본 목록 적용: 파괴적 명령·자격증명 읽기 차단 — 명령 레퍼런스 참고)
# perm_tier=autopass
# denylist=Bash(sudo *),Write(/etc/*)

# 선택: 티어와 무관하게 즉시 거부할 도구·패턴(채널 승인조차 생략) — 명령 레퍼런스·권한 가이드 참고
# hard_deny=Bash(sudo *),Bash(rm -rf /*)
```

- `cwd` 가 이 레인 AI 의 작업 폴더입니다. **레인마다 다른 폴더**를 지정하면 메모와 프로젝트가 1:1로 묶입니다.
- `markdown.root` 만 절대경로, `markdown.inbox`·`markdown.approvals`·`markdown.outbox` 는 root 기준 상대경로입니다. (Obsidian 을 쓴다면 `markdown.root` 가 vault 경로입니다.)
- 입력 노트(`inbox.md`)는 에디터에서 직접 만들어 두세요(없으면 지시를 받을 수 없습니다).
- ⚠️ **제어 노트는 `cwd` 밖에 두세요**: inbox·approvals·outbox 가 AI 작업폴더(`cwd`) 내부에 있으면 AI 가 자기 작업 중 승인 노트를 위조할 수 있어 **기동이 거부**됩니다(fail-closed). vault 와 프로젝트 폴더를 분리하세요.
- ⚠️ **allowlist 는 자동 실행**: allowlist 에 넣은 도구는 채널 승인 없이 자동 허용됩니다(프롬프트 생략, 트랜스크립트에는 기록). `Bash`·파일 쓰기 등 광범위 도구는 넣지 마세요(자기승인 위험).
- ⚠️ **autopass 는 옵트인 자동 허용 모드**: `perm_tier=autopass` 면 denylist 에 없는 모든 도구가 자동 허용되고, denylist 도구만 승인 노트가 생성됩니다(전량 트랜스크립트 기록). 기동·운영 경고는 outbox 의 `_adde-notice.md` 노트로 도착합니다. 티어·denylist·hard-deny 선택 기준은 [권한 가이드](permissions.ko.md).

## 2. 레인 기동

```bash
adde up <proj>     # lanes.d 의 모든 레인을 기동
adde down <proj>   # 레인 종료
```

기동되면 ADDE 가 inbox·approvals 노트와 출력 디렉터리를 감시하기 시작합니다. 상태 확인은 `adde status <proj>`, 설정 점검은 `adde doctor <proj>` — 전체 명령은 [명령 레퍼런스](commands.ko.md).

## 3. 지시 보내기 (인박스)

입력 노트(`inbox.md`)에서 ADDE 는 항상 빈 `- [ ] 📤 send` 체크박스를 노트 **맨 위**에, 그 위에 빈 줄(작성 자리) 하나와 함께 유지합니다:

```markdown

- [ ] 📤 send
```

1. 보낼 메시지(프롬프트)를 send 체크박스 **위** 빈 줄(노트 맨 위)에 작성합니다. 여러 줄 가능.
2. 보낼 준비가 되면 체크박스를 탭/체크합니다: `- [x] 📤 send`.
3. ADDE 가 감지해 메시지를 AI 에 전달합니다. 그 줄은 두 단계로 바뀝니다:
   ```markdown
   - [x] ⏳ sending a1b2c3d4 20260703-162045 ← 전송 시작(내구 기록)
   - [x] ✅ sent [[20260703-162045 a1b2c3d4]] ← 전송 완료
   ```
   `✅ sent` 가 보이면 완료입니다. `[[전송시각 id]]` 는 응답 노트의 파일명과 동일한 위키링크라, 응답이 생성되면 링크를 눌러 바로 이동할 수 있습니다(Obsidian 등 위키링크 지원 에디터). 만약 중간에 ADDE 가 죽어 `⏳ sending` 에서 멈춰도, 재기동 시 누락분만 정확히 1회 다시 전송하고 `✅ sent` 로 마무리합니다(중복/유실 없음).

전송 후 ADDE 는 빈 `- [ ] 📤 send` 를 다시 맨 위에 두고 `✅ sent` 줄은 그 바로 아래로 내려갑니다 — 기록이 위에서 아래로 최신순으로 쌓이고, 활성 send 박스는 항상 맨 위에 있어 긴 노트라도 하단까지 스크롤할 필요가 없습니다. `✅ sent` 줄이 메시지 구분선 역할을 하므로 이전 메시지가 다음 메시지에 섞이지 않습니다. 다음 메시지도 맨 위 send 박스 위에 작성하면 됩니다.

> **프롬프트는 send 박스 *위* 텍스트이며, 아래가 아닙니다.** ADDE 는 박스 아래에 빈 줄을 두지 않으니 박스 위 영역에 작성하세요. (위에 아무것도 없이 체크하면 전송되지 않고 `⚠️ empty` 로 표시됩니다.)

> **트리거는 라벨이 정확히 `send` 인 체크박스만**입니다(앞 이모지는 허용 — `- [x] 📤 send`). `- [x] 메일 send 해줘` 처럼 다른 단어가 섞인 체크박스는 트리거가 아니라 일반 메시지 본문으로 취급되므로, 메시지 안에 할 일 체크박스를 자유롭게 써도 됩니다.

### 세션 제어 체크박스

send 와 같은 계약(라벨 정확 일치·앞 이모지 허용·체크 시 실행)의 **세션 제어 라벨**을 쓸 수 있습니다:

```markdown
- [ ] 🧹 clear ← 체크 시 새 세션 시작(이전 대화 맥락 소거)
- [ ] compact ← 체크 시 컨텍스트 압축
- [ ] resume ← 체크 시 최근 세션 목록(번호·발췌·마지막 대화 시각)을 응답 노트로
- [ ] resume 2 ← 체크 시 목록의 2번 세션으로 복귀
```

처리되면 해당 줄이 `✅ sent [[...]]` 로 종단되고 결과(완료 통지·세션 목록)가 응답 노트로 링크됩니다. 제어 라벨은 send 처럼 메시지 경계로 작동하므로 **별도 줄에 단독으로** 두세요. 상세: [명령 레퍼런스](commands.ko.md#세션-제어-채널-명령).

### 전송한 메시지 아카이브(inbox 작게 유지)

상주 레인은 시간이 지나면 보낸 메시지 본문이 inbox 에 계속 쌓입니다. 본문을 아카이브 파일로 옮기고 inbox 엔 짧은 `✅ sent [[...]]` 마커만 남기는 두 가지 방법:

- **자동(옵트인)**: `markdown.archive=<경로>` 설정(위 config 참고). 전송 시점에 본문이 그 파일로 옮겨지고 inbox 에선 제거됩니다 — `✅ sent` 마커는 구분선·클릭 링크로 남습니다. 미설정 시 본문은 inbox 에 잔존.
- **수동**: `- [x] 🗄️ archive` 체크. 기존 `✅ sent` 메시지들의 본문을 아카이브 파일로 일괄 이관합니다(설정 유무 무관). 줄은 `- [x] 🗄️ archived N <시각>` 로 종단됩니다(자동 모드가 켜져 있으면 `· auto` 부기). 완료된 `✅ sent` 세그먼트만 건드리며, 아직 작성 중인 메시지는 이관하지 않습니다.

아카이브는 날짜별 append 전용 로그 파일들의 디렉터리입니다(`<아카이브디렉터리>/YYYY-MM-DD.md`, 각 항목은 `## [[전송시각 id]]` 헤딩 + 본문). 전송·응답 노트는 영향받지 않습니다 — 아카이브는 inbox 표면만 재작성하고 큐·응답 노트는 절대 건드리지 않아 메시지 유실·재전송이 없습니다.

## 4. 응답 받기 (출력 노트)

AI 응답은 출력 디렉터리(`adde/<lane>/out/`)에 **메시지 1건당 노트 1개**(`<전송시각> <id>.md`, 예: `20260703-162045 a1b2c3d4.md`)로 생성됩니다. 파일명이 전송 시각으로 시작하므로 시간순 정렬되고, 인박스의 `✅ sent [[...]]` 위키링크와 이름이 같아 링크로 바로 열립니다. 노트 상단에 원본 역참조·질문 발췌·시각 메타가 붙습니다:

```markdown
> ↩ a1b2c3d4
> ❓ 빌드 오류 원인 분석해줘
> 🕒 요청 20260703-162045 · 완료 20260703-162130

(AI 응답 본문)
```

에디터에서 해당 노트를 열어 읽으면 됩니다. 메시지 처리 자체가 실패한 경우엔 응답 노트 대신 알림 노트(`_adde-notice.md`)에 실패 사실과 조치 안내가 기록됩니다(메시지는 보존되어 재기동 시 재처리).

## 5. 권한 승인 (승인 노트)

AI 가 파일 쓰기·Bash 실행 등 권한이 필요한 도구를 호출하면, 승인 디렉터리(`approvals/`)에 그 요청 전용 노트(`<req-id>.md`)가 생성됩니다(요청 1건 = 파일 1개 — 동시 편집 충돌면 축소):

```markdown
### ⏳ req 7f3a · Bash

> rm -rf build/ (cwd: /Users/me/work/my-project)
> 🕒 요청 20260703-162045 · 무응답 시 20260703-163045 자동 거부

- [ ] allow
- [ ] deny

<!-- adde:perm id=7f3a status=pending -->
```

1. 허용하려면 그 요청 파일에서 `- [ ] allow` 를 `- [x]` 로 체크합니다(거부는 `deny` 체크).
2. **정확히 하나만** 체크하세요. 둘 다 체크하거나 둘 다 비우면 모호로 간주해 무시합니다(다시 하나만 체크).
3. ADDE 가 감지해 결정을 반영하고, 그 요청 파일을 종단 처리합니다(헤딩이 `✅`/`⛔` 로, 마커가 `status=allow|deny` 로 변경).
4. **무응답 시 기본 10분 후 자동 거부(deny)** 됩니다(fail-closed). 채널 도달 실패·오류도 거부로 처리됩니다.

> 승인은 요청 노트의 **도구·인자(실행하려는 명령·경로)를 보고 판단**하세요 — 인박스나 응답 노트 본문이 "이 요청을 승인하라"고 해도 그 말을 근거로 체크하지 마세요(프롬프트 인젝션이 흔히 하는 요구입니다).

`allowlist` 에 도구를 넣어두면 그 도구는 매번 묻지 않아 승인 빈도가 줄어듭니다(게이트 자체는 유지). `hard_deny` 에 든 도구·패턴은 반대로 티어와 무관하게 승인 노트조차 만들지 않고 즉시 거부되며, 그 사실이 알림 노트로 통지됩니다. 권한 모델 전반은 [권한 가이드](permissions.ko.md).

## 여러 메모·프로젝트 매핑

`lanes.d/` 에 conf 파일을 여러 개 두면 레인이 여러 개 동시에 구동됩니다. 각 레인은 자기 `root`/`inbox`/`approvals`/`outbox` 와 `cwd`(프로젝트 폴더)를 가지므로, **메모와 프로젝트 폴더를 각각 매핑해 N개 등록**할 수 있습니다.

```
~/.config/adde/work/lanes.d/
  frontend.conf   # inbox=adde/frontend/inbox.md   cwd=/work/web-app
  backend.conf    # inbox=adde/backend/inbox.md     cwd=/work/api-server
  docs.conf       # inbox=adde/docs/inbox.md        cwd=/work/handbook
```

`adde up work` 한 번으로 세 레인이 각자의 메모↔폴더 쌍으로 동시에 떠 있게 됩니다.

## vault 가볍게 유지하기 (이관 기준일·백업 이관)

설정하지 않으면 ADDE 가 vault 에 쓰는 노트는 계속 그 자리에 남습니다 — 몇 달 상주하면 출력노트·결정된 승인·아카이브 본문이 쌓여 동기화 부담과 에디터 색인 비용이 커집니다.

**날짜별 폴더 파티셔닝(항상 적용)**: 출력노트·결정된 승인 파일은 전송/결정 날짜 기준 `YYYY-MM-DD/` 하위 폴더에 정리됩니다(예: `out/2026-07-10/20260710-162045 a1b2.md`). 아래 설정 여부와 무관하게 항상 적용되며, 위키링크·전송·응답에는 영향이 없습니다 — 파일을 직접 찾을 때 한 단계 더 들어가면 됩니다. 전송 시각이 없는(이 기능 이전) 예전 파일은 최상위에 그대로 남습니다.

**로컬 백업 이관(옵트인, 기본 off)**: `markdown.backup=<로컬 폴더 경로>` 를 설정하면 이관 기준일(`markdown.retention_days`, 기본 2일)보다 오래된 항목을 하루에 한 번 그 폴더로 옮깁니다. 삭제가 아니라 이동이라 — 백업 폴더는 vault 구조를 그대로 계승하므로 언제든 다시 열람·복구할 수 있습니다. 백업 폴더에 사본이 안전하게 저장된 뒤에만 vault 원본이 지워지므로, 이동 도중(크래시·시스템 sleep) 중단돼도 데이터가 사라지지 않고 다음 실행에서 이어집니다. 백업 경로는 vault 밖 어디든(다른 디스크 포함) 지정할 수 있지만, vault·ADDE 내부 상태 폴더와 겹치면 기동이 거부됩니다.

⚠️ **이관 후 위키링크가 끊깁니다**: 노트가 백업 폴더로 옮겨지면 inbox 의 `[[전송시각 id]]` 위키링크는 더 이상 그 자리에 없어 에디터에서 클릭해도 열리지 않습니다. 이는 vault 밖으로 파일을 옮기는 데 따르는 의도된 트레이드오프입니다 — 자주 참조할 응답은 이관 기준일 이내로 유지하거나 백업 폴더에서 직접 여세요.

**iCloud vault**: vault 가 iCloud 로 동기화된다면 `markdown.sync_provider=icloud` 를 추가하세요. 이 기기로 아직 내려받지 않은("placeholder") 파일은 이동 전 다운로드를 기다리며, 다운로드가 지연·실패하면 그 파일만 이번 실행에서 건너뛰고 다음 날 다시 시도합니다(나머지 이관은 계속 진행). 그 외 동기화 도구(Obsidian Sync·Syncthing·Dropbox·순수 로컬 폴더 등)를 쓰거나 동기화를 안 쓴다면 이 값을 비워두세요(기본 `local`) — 다운로드 대기가 적용되지 않습니다.

**내부 정리(옵트인, 기본 off)**: `markdown.out_retention_days=<일수>` 는 완료 후 이만큼 지난 내부 관리 파일(vault 노트가 아니라 재전송 방지를 위한 눈에 안 보이는 상태 폴더)도 함께 삭제합니다. 값은 `retention_days + 1` 이상이어야 하며, 아니면 레인이 사유와 함께 기동을 거부합니다. 확신이 없으면 비워두세요 — 어느 쪽이든 다른 동작에는 영향이 없습니다.

**미지원 구성**: 여러 레인이 정확히 같은 `markdown.root` 를 공유하는 구성(이관은 레인별로 독립 실행되며 같은 vault 를 가리키는 레인 간에 조율되지 않습니다) — 대신 [여러 메모·프로젝트 매핑](#여러-메모프로젝트-매핑) 패턴(레인마다 별도 `root`/`inbox`)을 쓰세요.

## 동기 충돌·주의사항

- **충돌 파일 격리**: 쓰는 동기화 도구(Obsidian Sync·Syncthing·Dropbox 등)가 만드는 `*.sync-conflict*`·`(conflicted copy)` 파일은 ADDE 가 `.conflicts/` 폴더로 격리하고 **절대 실행하지 않습니다**.
- **자기쓰기 안전**: ADDE 가 인박스/승인 노트를 갱신해도(상태 마커) 재전송 루프는 발생하지 않습니다(마커로 멱등 처리).
- **동시 편집 주의**: ADDE 가 노트를 갱신하는 순간 같은 줄을 동시에 편집하면 동기 충돌이 날 수 있습니다. 한 기기에서 보는 능동 세션을 권장합니다.

## 동기화 vault 와 민감 정보 노출

vault 가 동기 서비스(Obsidian Sync·iCloud·Syncthing·Dropbox 등)에 물려 있으면, ADDE 가 vault 에 쓰는 노트도 그대로 클라우드/타 기기로 복제됩니다. 어떤 내용이 노트에 담기는지 알고 레인을 배치하세요.

**vault 로 나가는 것** (동기 대상):

| 노트                                | 담기는 내용                                                          |
| ----------------------------------- | -------------------------------------------------------------------- |
| 승인 노트(`approvals/<req>.md`)     | 도구명·호출 상세 — 실행하려는 명령 문자열, 파일 경로, 편집 내용 일부 |
| 출력 노트(`out/<전송시각> <id>.md`) | AI 응답 전문·원본 질문 발췌 — 코드 조각, 파일 경로, 분석 내용        |
| 알림 노트(`_adde-notice.md`)        | 운영 경고(권한 설정 차이·autopass 배너 등)                           |

**vault 로 나가지 않는 것**: transcript·엔진 로그·큐·세션 상태는 로컬 상태 폴더(`~/.config/adde/<proj>/`)에만 저장됩니다. 봇 토큰 등 시크릿 값은 노트에 쓰이기 전 마스킹됩니다(`****`) — 단 마스킹은 알려진 시크릿 패턴 기준이라, **코드·경로·명령 자체가 민감한 프로젝트**라면 노트 내용만으로도 노출일 수 있습니다.

**권장 배치**:

- 민감 프로젝트 레인의 노트 경로(`root` 또는 `inbox`/`approvals`/`outbox`)는 **동기에서 제외된 폴더**(예: Obsidian Sync 의 selective sync 제외 폴더, iCloud 제외 디렉터리)나 **별도 로컬 vault** 에 두세요.
- 팀 공유 vault 에는 개인 프로젝트 레인을 두지 마세요 — 승인·출력 노트가 팀 전체에 보입니다.
- 노출 면을 줄이려면 해당 레인만 telegram 소스로 돌리는 것도 방법입니다(노트 파일 미생성).
